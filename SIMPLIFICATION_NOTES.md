# Где можно упростить без вреда

Краткая техническая оценка по `be-btrfs.sh` с приоритетом на изменения без влияния на поведение.

## 1) Вынести единый парсер `fstab`-опций для `/`

Сейчас похожая логика проверки `subvol=` / `subvolid=` повторяется в:
- `check_mount_opts`
- `_check_clone_fstab`
- `cmd_check`

Безопасное упрощение: сделать маленький helper, который возвращает состояние (`ok`/`warn`/`error`) и найденные значения. Это:
- уберёт дублирование regex;
- снизит риск расхождения поведения между командами;
- упростит добавление новых правил проверки.

## 2) Унифицировать чтение полей из `btrfs subvolume list`

Парсинг `id` и `svname` через `awk` встречается в нескольких местах (`_iter_subvols`, `cmd_prune`, `_prune_legacy`, `cmd_clone`).

Безопасное упрощение: оставить один путь итерации (через `_iter_subvols`) и передавать callback-функции, где это возможно. Для мест с массивами можно добавить отдельный helper, который заполняет массивы, чтобы не дублировать `awk`-парсинг.

## 3) Убрать повтор вычисления "Creation time"

В `cmd_list` время создания берётся одинаково для BE и снапшотов. Можно вынести helper вида `_subvol_created_human <path>`, который вернёт форматированную строку или `?`.

Плюсы: меньше копипаста и проще менять формат вывода в одном месте.

## 4) Сгруппировать шаблон create/clone/snapshot

Команды `cmd_create`, `cmd_clone`, `cmd_snapshot` используют похожую схему:
1. валидация аргументов,
2. вычисление source,
3. создание subvolume,
4. запись metadata,
5. вывод статуса.

Безопасное упрощение: вынести общий "каркас" в приватные helper-функции и оставить в командах только разбор CLI-опций.

## 5) Снизить количество локальных вложенных callback-функций в `cmd_list`

Сейчас внутри `cmd_list` объявляется несколько локальных callback-функций. Это рабочий подход, но усложняет чтение.

Безопасное упрощение: часть callback-функций вынести в `private`-блок с параметрами (через глобальные/контекстные переменные или аргументы), чтобы сократить размер `cmd_list`.

---

## Что НЕ стоит упрощать "в лоб"

- Логику mount/unmount/chroot (`_with_mounted_be`, `_be_mount`, `_be_umount`): там уже учтены cleanup и DNS-нюансы.
- Ветвление в `cmd_prune`: правила с `min_keep` + `min_age` уже достаточно явно выражены; тут важнее аккуратные тесты, чем агрессивный рефакторинг.

## Рекомендуемый порядок рефакторинга

1. Пункт 3 (helper для Creation time) — самый безопасный.
2. Пункт 1 (fstab parser helper).
3. Пункт 2 (унификация `subvolume list` парсинга).
4. Пункты 4–5 (структурные улучшения команд).
